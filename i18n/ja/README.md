# Agent Capability Intake

[English](../../README.md) · [繁體中文](../zh-Hant/README.md) · [日本語](README.md)

> **手順書ではなく、判断の方向を示すためのものです。**
>
> このリポジトリは、人と AI エージェントが外部のエージェント機能を考えるためのモデルを提供します。すべての手順、実行環境、実装方法を指定するものではありません。まず [GUIDE.md](GUIDE.md) を読み、目の前の問いに必要な次の文書だけを読んでください。

これは、外部の AI エージェント機能をローカル環境に導入すべきか判断するための、エビデンス優先のガバナンスキットです。

スキル、プロンプト、フック、MCP サーバー、プラグイン、エージェント、CLI ラッパー、実行時パッケージは、**審査が済むまで信頼できない**ソースとして扱います。基本の流れは次のとおりです。

```text
external source -> digest -> absorb | no-absorb
```

## この流れが必要な理由

AI エージェント機能を導入すると、エージェントが取得できる情報、呼び出せるツール、実行できる操作が変わります。人気があること、十分なドキュメントがあること、デモで動いたことだけでは、有効化の根拠になりません。

このキットでは、混同されがちな三つの行為を分けます。

1. **digest（精査）**：出所とバージョンを確認し、内容、適合性、リスクを評価して、あとで参照できる結論を残します。この段階で実行権限は与えません。
2. **absorb（導入）**：承認した動作だけを指定したローカルの導入先に反映し、検証を行い、ロールバック手順を残します。
3. **no-absorb（不採用）**：導入しない理由、代替案、再検討する条件を明記します。これは未完了ではなく、完結した判断です。

## 守るべき原則

- digest では、ソースをインストール、有効化、実行してはいけません。
- absorb はローカル環境向けに必要な動作を再構成して導入する判断であり、アップストリームの指示をそのまま複製することではありません。
- 有効な機能には、ソース、責任者、リスク階層、検証の証拠、ロールバック手順または廃止手順が必要です。
- `no-absorb` の判断は見つけられる状態に保ち、同じソースを何度も調査しないようにします。
- 自動化は記録を検証できますが、ソースを勝手に有効な状態へ昇格させてはいけません。

## 機能の発見と階層的な読み込み

承認済みの機能でも、すべてのルールとリファレンスを各タスクで読み込めば、コンテキストの負担になります。機能をどう見つけ、いつ読み込むかも、有効化の設計に含めるべきです。

1. **常時適用される契約**：グローバルまたはリポジトリの指示は薄く保ちます。権限の境界、完了の基準、詳細を読む必要があるトリガーだけを書きます。
2. **機能カタログ**：機能が関連すると判断されるまで、スキルの名前と短い説明だけを公開します。説明はルーティングのためのインターフェースであり、規則集でも安全上の判断でもありません。
3. **選択されたスキル**：完全な `SKILL.md` は選択後にだけ読み込みます。そこにはワークフローの契約、安全上の境界、次に必要になり得る資料への直接の案内を記します。
4. **リファレンス**：明示されたトリガーが当てはまるときだけ、個別の参照資料を読みます。複数のスキルにまたがる方針は安定した共用の `references/` に置き、ワークフロー固有の詳細はスキルのそばに置きます。
5. **スクリプトとテンプレート**：選択されたスキルから使えるようにしますが、その内容を常時適用される指示として注入しません。

トップレベルの指示ファイルから、長いリファレンスの木を再帰的に取り込まないでください。一部の実行環境は取り込まれた指示ファイルを即座に連結するため、必要なときだけ読む設計が崩れます。関連性の判定でスキルを選ぶことはできますが、インストール、ツールの事前承認、有効化を許可するものではありません。

現在の実装には違いがありますが、[Codex](https://developers.openai.com/codex/codex-manual.md#customization-and-tooling)、[Claude Code](https://code.claude.com/docs/en/slash-commands)、[GitHub Copilot CLI](https://docs.github.com/en/copilot/how-tos/copilot-cli/customize-copilot/add-skills) は、機能カタログのメタデータ、選択された指示、必要時に読む資料を区別しています。[Gemini CLI](https://geminicli.com/docs/cli/gemini-md/) は対照例です。階層的なコンテキストファイルを各プロンプトで連結するため、この種のファイルは特に小さく保つ必要があります。

## はじめ方

1. [`templates/intake-record.md`](../../templates/intake-record.md) から導入審査の記録を作成します。
2. ソースを有効化せずに digest を完了します。
3. 対応を一つだけ選びます。
   - [`absorb`](../../templates/absorb-decision.md)
   - [`no-absorb`](../../templates/no-absorb-decision.md)
   - 利用を止める必要がある安全上の懸念がある場合は `quarantine`
4. `absorb` を選ぶ場合は、実行時環境を変更する前に合格／不合格の確認方法とロールバック手順を定義します。
5. 記録は、チームが正式な根拠として扱う機能レジストリまたは意思決定ログに保存します。

## 対象

エージェントの動作や権限に影響する外部カスタマイズを対象にします。

- スキル、プロンプト、指示、カスタムエージェント、テンプレート
- フック、プラグイン、拡張機能、CLI ラッパー、実行時パッケージ
- MCP サーバーとその他の連携ツール

このキットはパッケージマネージャー、実行時サンドボックス、セキュリティスキャナー、自動承認エンジンではありません。必要に応じて、プラットフォームの制御や CI と組み合わせてください。

## リポジトリ構成

- [`GUIDE.md`](GUIDE.md) — 人とエージェントが読む道筋を選ぶための簡潔な地図。
- [`SPEC.md`](../../SPEC.md) — 状態モデル、判定ゲート、最低限の証拠。
- [`schemas/intake-record.schema.json`](../../schemas/intake-record.schema.json) — 持ち運び可能で機械が読み取れる記録形式。
- [`templates/`](../../templates/) — 人がレビューするための Markdown 記録。
- [`examples/`](../../examples/) — 安全に導入する判断と、理由を残した不採用の例。
- [`i18n/`](../README.md) — 翻訳の配置と保守ルール。

## ステータス

これは初期段階のリファレンス実装です。スキーマとテンプレートは意図的に小さくしています。導入判断を明確にし、安全にし、あとから見直しやすくできるかについてのフィードバックを歓迎します。
